# 模型真实性验证项目说明（可直接放 GitHub）

## 项目目标

本项目用于审计第三方 API 服务商提供的模型是否“名实相符”。

典型场景：服务商声称是 `GPT-5.4`，但实际能力可能接近旧模型（例如 GPT-4 代）。

本项目通过**高压题库 + 百分制评分 + 多轮稳定性分析**输出“真实性分”，用于采购验收和纠纷留证。

---

## 目录结构

```text
model_verification/
├── questions.md            # 测试题库与执行说明
├── score_responses.py      # 自动评分脚本（百分制）
├── score_report.md         # 自动生成的评分报告
└── runs/
    ├── template.md         # 答案模板
    ├── <model>.md          # 某模型第1轮
    ├── <model>_v2.md       # 某模型第2轮
    └── <model>_v3.md       # 某模型第3轮
```

---

## 核心方法

1. **结构化身份声明检查**
   - 检查 `MODEL_NOTE` 与题内身份 JSON 是否一致。
2. **高难度任务能力检查（100分）**
   - 大整数算术、同余方程、代码推理、拓扑排序、最短路径、组合计数等。
3. **多轮稳定性检查**
   - 同一模型至少 3 轮，计算波动（标准差）。
4. **同源线索检查**
   - 不同轮次回答相似度，过高则标记疑似同源。

---

## 评分解释

- **任务分（100）**：Q1~Q10 加权得分。
- **轮次真实性分（100）**：任务分 + 身份一致性加分。
- **模型真实性总分（100）**：按模型聚合多轮分数后，叠加稳定性，扣除轮次不足惩罚。

可信区间：

- `>=95`：高度可信
- `90~94.9`：较高可信
- `80~89.9`：待进一步验证
- `70~79.9`：可疑
- `<70`：高风险（疑似虚标）

---

## 推荐执行流程

1. 在 Cursor 切到目标模型。
2. 让模型读取 `questions.md` 并写入 `runs/<model>.md`。
3. 同一模型重复 3 轮（`_v2`、`_v3`）。
4. 对多个模型重复。
5. 运行评分脚本，查看终端输出和 `score_report.md`。

---

## GitHub 使用建议

- 将 `model_verification/` 整个目录纳入仓库。
- 每次测试后提交一次，保留可审计历史。
- 建议提交信息包含日期、服务商、计费方案。

示例提交信息：

- `feat: add 2026-03-31 provider-A model authenticity benchmark runs`
- `chore: update score report for gpt-5.4-xhigh-fast v1~v3`

---

## 风险声明

本项目输出的是“行为真实性证据”，不是厂商后台内部模型 ID 证明。

但对采购验收来说，这种可复现、可量化、可对比的证据链通常已经足够用于判断“是否疑似虚标”。
